# Plan v1 — relay-patch

## Problem

Maintainer and Fork User sometimes disagree → unmerged PRs. Fork User wants both
(up-to-date official release) + (their custom features/bugfixes). Existing trade-off:
official release loses patches; forked release loses upstream updates.

## Solution

A workflow + tooling where **patches are intent, not diffs**, and an AI agent
re-realizes that intent against every upstream release. The canonical artifact is
a public `USERNAME/.relay-patch` repo of intent files; `relay-patch/main` (the
buildable fork branch) is a derived output, never hand-edited.

Core invariant: **intent is truth, diffs are evidence.** Everything else follows.

---

## Repository layout — `USERNAME/.relay-patch`

One repo per user, holds patches for many upstream targets.

```
.relay-patch/
├── README.md                       auto-generated human-browsable index
├── global.json                     user identity, default agent, schedule
└── repos/
    └── github.com/owner/repo/      namespaced by full host/owner/repo
        ├── upstream.json           upstream main SHA tracking, sync schedule
        ├── manifest.json           patch list, apply order, status
        └── patches/
            └── <patch-id>/
                ├── INTENT.md       source of truth (natural language)
                ├── ACCEPTANCE.md   machine-checkable criteria where possible
                ├── reference.diff  last successful realization (evidence)
                └── tests/          optional runnable acceptance tests
```

Why `github.com/owner/repo` and not `owner-repo` slug:
- Disambiguates GitHub vs GitLab vs self-hosted.
- Survives same-name collisions across hosts.
- Mirrors a clone URL — zero ambiguity on import.

### Patch IDs

`<kebab-slug>-<ULID8>` — e.g. `dark-mode-toggle-01j6q3f8`.

- **Slug** = human-readable, may change as intent evolves.
- **ULID8** = forever-stable uniqueness, sortable by creation time.
- Renames update slug; ULID stays. Alias map in `manifest.json` tracks old slugs.
- Cross-user import never clashes (different author, same ULID8 ≈ impossible).

Folder name = full ID. Title in `INTENT.md` frontmatter.

### `INTENT.md` frontmatter schema

```yaml
---
id: dark-mode-toggle-01j6q3f8
title: Add dark mode toggle in settings
target_repo: github.com/owner/repo
target_area: [src/settings/, src/theme/]
status: applied               # drafting | applied | drifted | re-deriving
                              # | needs_human | upstreamed | retired
applied_upstream_pr: none      # none | open(url, sha) | merged(sha, release)
                              # | closed(reason)
version: 2                    # bumps when author evolves intent
license: MIT
author: USERNAME              # original author (preserved on import)
source_url: null              # set when imported from another user
imported_at: null
created: 2026-06-13
last_realized_against_commit: abc123
verifies_with: bun test patches/dark-mode-toggle-01j6q3f8/tests/
---

## Intent
...

## Why
...

## Non-negotiables
...

## Implementation notes
...  (includes prior drift history — what upstream changed and how we adapted)
```

### `manifest.json`

```json
{
  "target_repo": "github.com/owner/repo",
  "upstream_main_branch": "main",
  "apply_order": [
    "fix-login-redirect-01j5xa2z",
    "dark-mode-toggle-01j6q3f8"
  ],
  "patches": {
    "dark-mode-toggle-01j6q3f8": {
      "status": "applied",
      "version": 2,
      "last_realized_against_commit": "abc123"
    }
  },
  "slug_aliases": {
    "dark-mode": "dark-mode-toggle-01j6q3f8"
  }
}
```

Apply order is first-class: patches may depend on each other.

## Branch model

| Branch | Owner | Mutated by | Purpose |
|---|---|---|---|
| `main` (upstream) | Maintainer | Maintainer | Source of truth for original code |
| `*` (patch scratch) | User + AI | User + AI | DRAFTING iteration for one intent |
| `relay-patch/*` | Orchestrator + AI | Re-derivation pipeline | Combining all patches against new upstream |
| `relay-patch/main` | Orchestrator | Orchestrator only | **Derived output**, never hand-edited |

Hard rule: **only the orchestrator writes to `relay-patch/main`.** To change
behavior, change the intent, then trigger re-derivation. This is the same
discipline as "never edit generated code."

History model: **hybrid.** `.relay-patch` is permanent source of truth;
`relay-patch/main` is a derived artifact, rebuilt (force-pushed) on every
re-derivation. Consumers treat it as a *release*, not a branch to collaborate on.

## Patch lifecycle — state machine

```
                            /relay-patch INTENT
                                     │
                                     ▼
                              ┌─────────────┐
                              │   DRAFTING  │◄────┐
                              │ (branch: *) │     │ /relay-patch INTENT
                              │ AI implements│    │ (iterate, same intent)
                              │ user tries  │     │
                              └─────┬───────┘     │
                          satisfied?├──no─────────┘
                                    │ yes
                                    ▼
                              ┌─────────────┐  user opens PR upstream
                              │   APPLIED   │──────────────┐
                              │ realization │              │
                              │ in relay-   │              ▼
                              │ patch/main  │       (attribute only:
                              └─────┬───────┘        applied_upstream_pr)
                    upstream drift │                     │
                           detected│                     │
                                    ▼                     │
                              ┌─────────────┐             │
                              │   DRIFTED   │             │
                              └─────┬───────┘             │
                                    │ scheduled           │
                                    ▼                     │
                              ┌─────────────┐             │
                              │ RE-DERIVING │             │
                              └─────┬───────┘             │
                       verify gate │                     │
                  ┌────────────────┼────────────────┐    │
                  ▼                ▼                ▼    │
            ┌──────────┐    ┌─────────────┐  ┌────────┐  │
            │ APPLIED  │    │ NEEDS_HUMAN │  │UPSTREAM│  │
            │ (re-     │    │ (blocked)   │  │_ED     │  │
            │  applied)│    └──────┬──────┘  └───┬────┘  │
            └──────────┘           │ user fixes    │      │
                                   └───────────────┼──────┘
                                                   ▼
                                             ┌──────────┐
                                             │ RETIRED  │
                                             └──────────┘
```

### States

| State | Meaning | Who acts |
|---|---|---|
| **DRAFTING** | `*` branch open, AI implementing, user testing. Intent file not yet committed. | User + AI |
| **APPLIED** | Intent committed to `.relay-patch`; realization merged into `relay-patch/main`. `last_realized_against_commit` recorded. | Orchestrator |
| **DRIFTED** | Upstream advanced past recorded commit. Passive flag — patch *might* still apply, we don't know yet. | System |
| **RE-DERIVING** | AI running in `relay-patch/*` against new upstream. | AI |
| **NEEDS_HUMAN** | Verification gate failed and AI retries exhausted (3×). **Blocks `relay-patch/main` from advancing.** | User |
| **UPSTREAMED** | Upstream now natively satisfies intent (via PR merge or independent implementation). | AI |
| **RETIRED** | Intent kept for history/sharing; no longer applied. | System |

### PR-to-upstream is an attribute, not a state

A patch is APPLIED regardless of PR status — your fork keeps the patch either way.
The `applied_upstream_pr` field on APPLIED transitions:

```
none ──(user opens PR)──► open(url, sha)
                          │
              ┌───────────┴────────────┐
              ▼                        ▼
        merged(sha, release)    closed(reason)
              │                        │
              │                        └─► back to APPLIED (patch lives on)
              ▼
        triggers UPSTREAMED on next re-derivation
```

UPSTREAMED triggers from either:
1. `applied_upstream_pr = merged(...)`, or
2. AI semantic check detects the intent is now natively satisfied upstream
   (handles the case where maintainer independently implemented the same thing).

### Transitions

| From → To | Trigger | Actor |
|---|---|---|
| DRAFTING → DRAFTING | `/relay-patch INTENT` again (iterate) | User |
| DRAFTING → APPLIED | `/relay-patch satisfied` | User |
| APPLIED → DRIFTED | scheduled poll: upstream SHA ≠ recorded | System |
| DRIFTED → RE-DERIVING | scheduled re-derive tick | System |
| RE-DERIVING → APPLIED | verification gate green | AI |
| RE-DERIVING → NEEDS_HUMAN | gate red, 3 retries exhausted | AI |
| RE-DERIVING → UPSTREAMED | AI detects intent satisfied upstream | AI |
| NEEDS_HUMAN → DRAFTING | user invokes `/relay-patch <id>` to fix | User |
| UPSTREAMED → RETIRED | confirmed stable upstream for N releases | System |

## The `DRIFTED → RE-DERIVED` AI contract

The heart of the project. Get this right, everything else is plumbing.

### Inputs

1. **Intent file** — natural language: what user wants, why, acceptance criteria,
   prior drift history.
2. **Previous realization** (`reference.diff`) — explicitly framed as
   *"reference, not source of truth."*
3. **Previous upstream baseline SHA** — so AI can run
   `git log <baseline>..upstream/main -- <target_area>` and see exactly what
   changed.
4. **Target codebase** — fresh `upstream/main` in `relay-patch/*`.
5. **Sibling context** — other APPLIED patches' realizations, so AI doesn't
   duplicate effort.
6. **Verification commands** — from INTENT.md frontmatter.

### Outputs

1. **New realization** — clean commits on `relay-patch/*` against new upstream.
2. **Drift report** — structured: what upstream changed, whether approach
   diverged from previous realization and why.
3. **Self-evaluation** — pass/fail per acceptance criterion.
4. **Confidence + `needs_human` flag.**

### Verification gate (all green to promote to APPLIED)

```
✓ Build succeeds
✓ Tests pass (patch's tests/ if defined, else repo default suite)
✓ AI self-eval: every acceptance criterion = pass (no "unknown" for auto-promote)
✓ Diff sanity: non-trivial vs upstream, not catastrophically large
✓ (optional) Idempotency: re-run produces same realization
```

**Failure policy:** never auto-ship yellow. "Unknown" on any criterion or
3 failed retries → NEEDS_HUMAN. This is the discipline that prevents silent
drift — the thing that would kill trust in the tool.

### Three interesting failure modes

1. **Upstream deleted what patch relied on.** AI must decide: re-target, or
   propose RETIRED. Surface to user either way — never auto-re-target silently.
2. **Upstream independently implemented same intent.** AI detects via semantic
   similarity between intended realization and upstream diff → UPSTREAMED. This
   is the *happy* failure — the patch won.
3. **AI confident but wrong.** Mitigations: idempotency check, diff-size sanity,
   strict acceptance-criterion evaluation. Also: **AI never updates
   `last_realized_against_commit`** — only the orchestrator does, after the
   gate passes.

## Patch discovery & reuse (Patch Directory)

`.relay-patch` is public → browsing = GitHub search today, aggregator site later.

### Import flow

```bash
relay-patch import https://github.com/ALICE/.relay-patch/blob/main/repos/\
github.com/owner/repo/patches/dark-mode-toggle-01j6q3f8/INTENT.md
```

1. Fetch INTENT.md + ACCEPTANCE.md + tests/.
2. Copy into your `.relay-patch` under same `<patch-id>`.
3. Preserve `author: ALICE`, set `source_url`, `imported_at: now`.
4. AI re-derives `reference.diff` against *your* repo's `target_area` (paths may
   differ from author's).
5. If `<patch-id>` already exists locally: refuse without `--force` (treats as
   update from upstream author).

### Cross-repo adaptation

If importing to a different repo than the author's, AI re-derives purely from
intent — `reference.diff` is hints only. This is where "intent is truth" pays
off: a well-written `dark-mode-toggle` intent can adapt to any React app.

### Updates from original author

- Author bumps `version` in INTENT.md and pushes to their `.relay-patch`.
- Subscribers see via git history of `source_url` (future: directory notifies).
- Subscriber decides when to pull update and re-trigger re-derivation.
- Patch versioning = simple integer bumps. SemVer is overkill for intent.

### Licensing

Patches are creative work. Default MIT; per-patch override in frontmatter.
Import preserves original license — never strip attribution.

## Scenarios

### Scenario 1 — PR merged upstream (normal)

1. Maintainer owns `owner/repo`, publishes releases from `main`.
2. User forks, clones locally.
3. User runs `/relay-patch "add dark mode toggle"`.
4. AI opens `*` branch, implements, user tests.
5. User iterates with `/relay-patch "..."` until satisfied.
6. User runs `/relay-patch satisfied`:
   - Intent written to user's `.relay-patch`.
   - Realization merged into user's `relay-patch/main`.
7. User opens PR upstream from `*` → `owner/repo#main`.
8. Maintainer merges PR.
9. On next scheduled poll, `relay-patch` detects intent natively satisfied
   upstream → UPSTREAMED → RETIRED. User now consumes upstream unmodified.

### Scenario 2 — PR rejected / not merged (relay-patch's reason to exist)

1. Maintainer owns `owner/repo`, publishes from `main`.
2. User forks, clones locally.
3. User runs `/relay-patch "add dark mode toggle"`.
4. AI opens `*`, implements, user tests, iterates.
5. User runs `/relay-patch satisfied`:
   - Intent → `.relay-patch`.
   - Realization → `relay-patch/main`.
6. User opens PR upstream; maintainer closes it ("not in scope").
   - `applied_upstream_pr = closed("out of scope")`.
   - Patch stays APPLIED. User keeps using `relay-patch/main`.
7. **Time passes. Maintainer publishes v2.0 of `owner/repo`.**
8. Scheduled poll detects drift: APPLIED → DRIFTED.
9. Orchestrator opens `relay-patch/*`, dispatches AI agent with intent + previous
   realization + drift report.
10. AI re-derives against v2.0, runs verification gate.
    - Green → RE-DERIVING → APPLIED. `relay-patch/main` force-pushed with new
      realization. User pulls the new build, still has dark mode + v2.0 features.
    - Red → NEEDS_HUMAN. User runs `/relay-patch dark-mode-toggle-01j6q3f8` to
      unblock in DRAFTING, then satisfied again.
11. Loop continues forever — user always has up-to-date upstream + their patch.

### Scenario 3 — Reusing someone else's patch

1. User B wants dark mode on the same repo.
2. B browses `github.com/ALICE/.relay-patch`, finds `dark-mode-toggle-01j6q3f8`.
3. B runs `relay-patch import <url>`.
4. AI re-derives against B's fork, preserving `author: ALICE`.
5. B's `relay-patch/main` now has dark mode — without B writing any intent.
6. When upstream drifts, *both* A and B get independent re-derivations. They may
   diverge if A's and B's forks differ, but both stay true to ALICE's intent.

## Open questions (not blocking v1 implementation)

- **Multi-patch interaction.** If patch A and patch B both touch `theme.tsx`,
  does the AI re-derive them sequentially (A then B) or jointly? Sequential is
  simpler; joint is more correct. Start sequential, surface conflicts.
- **Apply order changes.** What if A depends on B but `manifest.json` order is
  wrong? Detect via re-derivation failure, not by static analysis.
- **AI determinism.** Same intent + same upstream should yield same realization.
  Idempotency check catches violations; flag for human review.
- **Cost.** Periodic re-derivation uses AI tokens. Schedule must be tunable per
  repo (daily? weekly? on upstream release only?).
- **What if upstream deletes the entire `target_area`?** AI proposes RETIRED;
  user confirms. Never auto-retire.
- **Private upstreams.** `.relay-patch` is public — does intent leak info about
  private target repos? Document: don't put proprietary code in INTENT.md.

---

## Non-goals (explicit)

- Not a replacement for `git rebase`. No general-purpose patch management.
- Not a CI system. Verification runs locally or via scheduled orchestrator, not
  on every push.
- Not multi-host in v1. GitHub only; design accommodates others later.
- Not a package manager. Patches aren't installed into `node_modules`; they
  realize into source.
