# Plan

## Problem Focus

Maintainer and Fork User sometimes have different vision, resulted on unmerged PR, but Fork User often want their feature/bugfix + up-to-date official release. If they use the official release, they get the up-to-date release, but losing their feature/bugfix. If they use their forked release, they get their feature/bugfix, but losing up-to-date official release.

## Proposed Solution

A workflow to always have up-to-date official release combined with user's custom requirements.

## Scenarios (Work In Progress)

### Scenario 1: User submits PR and maintainer merge PR (full original repo code) - normal

1. Maintainer maintain code and use usual branches `main`,
2. User can fork
3. User can clone and run locally
4. User create new branch and can submit PR of that new branch to main original repo to maintainer
5. User can use the patched-forked version while waiting for PR to be merged
6. AI helps make sure PR and forked version is
5. Maintainer can review and merge PR
6. Maintainer can publish new version to GH for normal user
7. User tell `relay-patch` to update to latest version
8. `relay-patch` detect no patch different, it update to the official release

### Scenario 2: User submits PR and maintainer does not merge PR (full forked repo code) - relay-patch

1. Maintainer maintain code and publish to GH
2. User can fork
3. User can clone and run locally
4. User can submit PR to maintainer

## Important Branches

- `main`: the main branch of the original repo, maintained by the maintainer, it can be `main`, `master`, or any other name, but we will refer to it as `main` for simplicity. This branch is used by normal users and is the source of truth for the original code.
- `relay-patch/main`: the final result of combining up-to-date `main` and all user's patches
- `relay-patch/*`: the source for the PR to relay-patch branch, this is exclusive place for AI to progressively prepare all patches before merged to `relay-patch/main`
- `*`: the source for the PR to original repo

## Potential Features

- Patch Directory: Place for users to share, search, and re-use other user's patch
- Patch Management: Manage multiple patches across different repositories
